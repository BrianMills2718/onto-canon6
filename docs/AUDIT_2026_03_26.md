## 2. AMBIGUITIES

### 2.1 Where does extraction "eventually live"?

Parity matrix uncertainty #3: "CLAUDE.md and ECOSYSTEM_STATUS.md say extraction
'will move out' of onto-canon6 once stabilized. No destination is named."

CLAUDE.md's composability principle says "Extraction is a producer, not core."
But the extraction pipeline is deeply integrated: `pipeline/text_extraction.py`
(863 lines), `pipeline/service.py`, 14 prompt templates, progressive extraction
with 3 passes. This is NOT a thin adapter — it's a major subsystem.

**Recommendation**: Drop the "will move out" language. Extraction is a
first-class onto-canon6 capability. The composability principle should mean
"other producers can also feed assertions" (which is already true via
import-research-v3), not "extraction will be extracted."

### 2.2 ProbLog adapter placement contradicts FRAMEWORK.md

FRAMEWORK.md places the rule engine in Operations (llm_client). But we built
`problog_adapter.py` inside onto-canon6. The Gap 8 spike says "Placement:
onto-canon6 owns the fact-store adapter" but the evaluate-rules CLI is in
onto-canon6 too, not llm_client.

**Recommendation**: This is fine for now — the adapter loads facts from
onto-canon6's DB, so it belongs here. The FRAMEWORK can be updated to say
"fact-store adapters live with the data, rule evaluation can live anywhere."
Don't move code to match a doc — update the doc to match reality.

### 2.3 "Schema stability" undefined

Parity matrix uncertainty #6 proposes "no breaking changes to promoted graph
schema for 30 days" but this was never adopted or enforced. The schema has
been stable since Phase 11 (2026-03-18), which is 8 days ago.

**Recommendation**: Adopt the 30-day definition. Start the clock from the
last promoted_graph schema change.

### 2.4 OpenClaw success criteria untested

`.openclaw/success-criteria.yaml` and `mission-spec.yaml` exist but have
never been validated against the actual mission runner. They reference
`make extract` and `make summary` which work, but the review gate evaluation

## 4. RISKS & CONCERNS

### 4.1 Extraction quality is genuinely poor

The experiment showed 25% success rate across variants (5/20 pairs scored > 0).
The operational prompt works for simple extractions but the experiment variants
fail on complex multi-role assertions. The 37.5% PSYOP Stage 1 acceptance rate
hasn't improved.

**Risk**: If extraction quality doesn't improve, the entire pipeline produces
low-value output that wastes reviewer time.

**Mitigation**: The operational prompt works for new domains (financial,
academic, DoDAF). Focus on rebuilding experiment variants from the operational
prompt rather than developing new variants independently.

### 4.2 API reliability is a hard blocker for experiments

55% timeout rate via OpenRouter/litellm on the extraction experiment. This
makes systematic prompt comparison unreliable.

**Risk**: Can't run meaningful experiments if more than half the calls timeout.

**Mitigation**: Switch to direct Gemini API (bypass OpenRouter). Or use a
model with better availability. The `model_override` config already supports
this.

### 4.3 No automated quality gate on extraction

The pipeline extracts, validates structurally, and offers for review — but
there's no automated quality assessment. The LLM-judge (reasonableness scorer)
exists but isn't wired into the operational path.

**Risk**: Bad extractions consume reviewer attention without filtering.

**Mitigation**: Wire the LLM-judge into the extraction pipeline as a
post-extraction filter. Reject candidates below a reasonableness threshold
before they reach the review queue.

### 4.4 auto_resolution.py is exact-match only

The entity resolution uses exact case-insensitive name matching. This will miss
obvious matches like "4th PSYOP Group" vs "4th POG(A)" or "Gen. Holland" vs
"General Charles R. Holland."

**Risk**: Cross-chunk entity resolution will have high false-negative rate on
real data with name variation.

**Mitigation**: Add fuzzy matching (Levenshtein distance, or simple
normalization heuristics: strip titles, expand abbreviations). This is a known
Gap 3 limitation.

## 5. HAND-ROLLED vs OFF-THE-SHELF

### 5.1 ProbLog — CORRECT DECISION

ProbLog is the right choice. It's a mature library (2.2.10, pure Python, 2MB),
handles probabilistic inference natively, and avoids building a custom Datalog
interpreter. Good call.

### 5.2 Entity resolution — SHOULD USE A LIBRARY (eventually)

The current exact-name matcher is ~180 lines. For production fuzzy matching,
consider:
- **rapidfuzz** (already in Digimon's deps): Levenshtein, partial ratio, token sort
- **dedupe** (entity resolution library): ML-based record matching
- **splink** (Spark-based): probabilistic record linking

For now, exact match is fine. When fuzzy matching is needed, use rapidfuzz
(already available in the ecosystem) rather than hand-rolling distance metrics.

### 5.3 Extraction pipeline — CORRECTLY HAND-ROLLED

The extraction pipeline is custom but necessarily so — it's tightly coupled to
the ontology pack system, evidence span resolution, and review workflow. No
off-the-shelf extraction framework (LlamaIndex, Haystack, REBEL) would handle
the pack-driven validation and review lifecycle.

### 5.4 Ontology runtime — CORRECTLY HAND-ROLLED

The pack/profile/validation system is domain-specific and tightly integrated.
OWL/Protégé would be heavier, less observable, and harder to integrate with
the LLM extraction path.

### 5.5 Foundation IR — CONSIDER ADOPTING FtM

The Foundation Assertion IR is a custom spec. FollowTheMoney (FtM) is already
used by research_v3 and provides typed entity schemas, relationship modeling,
and serialization. The FtM → onto-canon6 mapping in `research_v3_import.py`
adds a translation layer that might not be needed if onto-canon6 adopted FtM
natively for entity typing.

**However**: FtM schemas are entity-focused, not assertion/predicate-focused.
onto-canon6's n-ary assertion model (predicate + roles) is richer than FtM's
binary entity-relationship model. Adopting FtM for entity types while keeping
the assertion model is reasonable.

**Recommendation**: Don't adopt FtM internally. The current translation layer
is thin enough. FtM is for entities; onto-canon6 is for assertions.
