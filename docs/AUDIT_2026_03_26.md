# Documentation & Planning Audit — 2026-03-26

Methodical review of `onto-canon6` documentation and active plans against:

- `project-meta/vision/FRAMEWORK.md`
- `project-meta/vision/FOUNDATION.md`
- `project-meta/vision/ECOSYSTEM_STATUS.md`
- `project-meta/vision/ROADMAP.md`
- current repo state as of 2026-03-26

## Critical Findings

### 1. Status authority was fragmented and internally contradictory

Before this audit, the repo simultaneously claimed:

- Plan 0020 was fully closed
- OpenClaw autonomous operation was complete
- DIGIMON consumer flow was proven end to end
- consumer-side adoption and end-to-end query flow were still unproven

The worst contradiction was `docs/STATUS.md`, which mixed real proofs with
overclaims about OpenClaw and consumer integration. The highest-authority repo
guidance (`CLAUDE.md`, `AGENTS.md`, `HANDOFF.md`) also asserted unconditional
closure of all 10 vision gaps.

**Disposition**: corrected the high-authority docs so they now distinguish:

- closed
- partially closed
- adapter-level proof
- spec drafted but runtime unproven

### 2. Historical and active plans were not clearly separated

`docs/plans/0001_successor_roadmap.md` still advertised itself as active even
though it is a completed bootstrap record. That made the plan surface harder to
scan and obscured which documents now govern active work.

**Disposition**: the roadmap is now marked as a completed historical baseline,
and the repo points readers toward Plans `0014`, `0019`, and `0020` for current
post-bootstrap work.

### 3. Gap 10 was overclaimed

Repo-local `.openclaw` mission files exist, but runtime proof was missing.
`project-meta/ops/openclaw/README.md` documents a success-criteria contract, but
`project-meta/ops/openclaw/run_task.py` currently contains no direct references
to `.openclaw/success-criteria.yaml` or `mission-spec.yaml`.

**Conclusion**: Gap 10 is specification-level partial progress, not a completed
autonomous-operations proof.

### 4. Gap 5 is stronger than previously documented, but not fully closed

The DIGIMON bridge is now stronger than the plan said:

- real export from `review_combined.sqlite3`
- real DIGIMON import into GraphML
- real DIGIMON operator query against imported graph

But the last semantic question remains open:

- confidence/weight handling on a non-unity slice is not yet validated

**Conclusion**: Gap 5 should be tracked as partially closed, not open and not
fully closed.

## Ambiguities

### 1. Consumer-side adoption vs repo-local proof

The adapters are real. The repo-local proofs are real. But neither DIGIMON nor
research_v3 is yet documented as using these paths in their normal workflow.

This distinction matters. "Adapter exists" is not "consumer integration is
done."

### 2. OpenClaw review gate semantics remain underspecified

The current repo-local mission specs define measurable thresholds, which is
good. But the exact review-evaluation path for extraction results is still only
documented, not exercised. The repo should not claim mission-level autonomy
until this is run for real.

### 3. Extraction packaging boundary is still a long-term architectural question

The previous docs implied extraction might eventually move out of onto-canon6,
but without naming a destination or trigger. That was too vague to guide
planning.

Current audit view:

- extraction is a first-class onto-canon6 producer today
- no move should be implied until there is a concrete shared boundary and a real
  maintenance or reuse reason

### 4. Schema-stability language is still softer than it should be

The ecosystem convergence docs talk about schema stabilization, but the
criterion is still not crisp enough. A concrete definition such as "no breaking
changes to promoted-graph/export contracts for 30 days" would be more useful
than the current hand-wavy wording.

## Clutter and Non-Compendiousness

### 1. Too many documents were trying to be "the current status"

The main clutter source was not raw volume. It was competing authority:

- `README.md`
- `docs/SUCCESSOR_CHARTER.md`
- `docs/STATUS.md`
- `docs/plans/0001_successor_roadmap.md`
- `docs/plans/0005_v1_capability_parity_matrix.md`
- `docs/plans/0020_vision_gap_closure.md`
- `HANDOFF.md`
- `CLAUDE.md`
- `AGENTS.md`

The cleanup in this audit was aimed at hierarchy, not deletion:

- charter = strategic anchor
- `0001` = historical bootstrap record
- `0005` = long-term capability ledger
- `0020` = active gap-closure tracker
- `STATUS.md` = concise proof snapshot
- `HANDOFF.md` = next-agent operational context

### 2. Some docs mixed plan, execution log, and status ledger

Plan 0020 had drifted into all three at once. That made it easy to append
"completed" blocks without reconciling the rest of the gap text.

That pattern is risky. The plan should remain the tracker; deep execution notes
belong in handoff or run artifacts.

## Risks

### 1. Extraction quality is still the dominant product risk

The architecture is largely not the blocker. The extraction quality and
experiment reliability remain the main risk to user value.

### 2. Consumer integration can still fail at the last mile

Repo-local proofs show the contracts are credible. They do not prove the
consumers will adopt them with acceptable ergonomics, dependency shape, or
operational semantics.

### 3. Exact-name entity resolution is too weak for cross-investigation use

Exact-name resolution was enough to prove the infrastructure. It is not enough
for real cross-document or cross-investigation normalization.

## Hand-Rolled vs Off-The-Shelf

### Keep hand-rolled

- The governed assertion / promoted-graph / review pipeline shape is
  ecosystem-specific and correctly custom.
- The DIGIMON import bridge is thin glue code and appropriate to hand-roll.
- The Foundation Assertion IR export layer is a boundary adapter and appropriate
  to hand-roll.

### Use off-the-shelf

- **ProbLog**: correct choice. The spike shows no reason to hand-roll a rule
  engine.
- **Fuzzy entity resolution**: do not hand-roll edit distance or token scoring.
  Use `rapidfuzz` when extending beyond exact-match resolution.

### Speculative hand-rolling to avoid

- A custom OpenClaw review-gate evaluator before proving the existing runner
  contract actually consumes the repo-local mission files.
- Any homegrown fuzzy-merge system more elaborate than a thin `rapidfuzz`-based
  policy layer.

## Recommendations

### Most valuable next review/planning steps

1. Prove consumer-side adoption paths in the consumer repos, not only in
   onto-canon6.
2. Run a real OpenClaw mission against onto-canon6 and treat that as the only
   valid Gap 10 closure evidence.
3. Add one non-unity-confidence DIGIMON validation slice and close the remaining
   Gap 5 semantic question.
4. Define an explicit schema-stability gate for cross-project contracts.
5. Add a library-backed fuzzy resolution mode with clear false-positive
   safeguards.

### Documentation/process recommendations

1. Keep `STATUS.md` narrow. It should summarize proofs, not speculate.
2. Keep `0020` as the active tracker until every gap is either closed or
   intentionally deferred.
3. Do not mark a gap complete merely because spec files or adapters exist.
   Require runtime evidence at the consumer or orchestration boundary.
4. Treat `HANDOFF.md` as operational truth for the next agent; keep it short and
   exact.

## Verdict

### Do we have clear unambiguous and compendious long-term plans?

Not fully, but the situation is now materially better after this cleanup.

What is clear:

- the strategic direction
- the bootstrap history
- the long-term capability ledger
- the active gap-closure tracker

What is still not fully clear:

- the exact closure standard for cross-project "integration complete"
- the schema-stability gate
- the OpenClaw runtime proof standard
- the future packaging boundary for extraction, if it ever moves

So the long-term plan is now mostly coherent, but not yet fully unambiguous.
The remaining ambiguity is concentrated in cross-project operationalization, not
in the core onto-canon6 architecture.
